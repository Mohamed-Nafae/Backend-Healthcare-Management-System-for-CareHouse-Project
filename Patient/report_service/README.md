# Report Service

A microservice for managing medical reports in the CareHouse patient management system. This service handles creation, retrieval, and deletion of clinical reports generated by doctors for patients, serving as a crucial component of the electronic health record (EHR) system.

## Overview

The Report Service is a RESTful Node.js/Express microservice that provides APIs for:
- **Report Management**: Create, read, and delete medical reports
- **Patient-Specific Reports**: Retrieve all reports for a specific patient
- **Doctor-Report Association**: Link medical reports to the treating doctor
- **Report Content Storage**: Store and reference detailed report content
- **Duplicate Prevention**: Prevent duplicate report content in the system
- **Clinical Documentation**: Maintain comprehensive medical report archives

## Installation

### Prerequisites
- Node.js (v14+)
- MongoDB running locally or remote connection
- npm or yarn

### Setup

1. Clone or navigate to the service directory
```bash
cd report_service
```

2. Install dependencies
```bash
npm install
```

3. Create a `.env` file in the root directory
```env
# MongoDB Connection
DATABASE_URI=mongodb://localhost:27017/reports
# or for remote MongoDB
DATABASE_URI=mongodb+srv://<username>:<password>@<cluster>.mongodb.net/reports

```

4. Start the service
```bash
# Development mode with auto-reload
npm run dev

# Production mode
npm start
```

The service will listen on `http://localhost:5005`

## Database Model

### Report

```javascript
{
  createdAt: Date,                   // Creation timestamp (auto-generated)
  patient: ObjectId,                 // Reference to patient (required)
  doctor: ObjectId,                  // Reference to doctor (required)
  report_content: ObjectId,          // Reference to report content (required, unique)
}
```

**Constraints:**
- `report_content` must be unique (prevents duplicate reports)
- Both patient and doctor are required fields
- All references use MongoDB ObjectIds for referential integrity

## API Endpoints

### Base URL
```
http://localhost:5005/api/reports
```

### Reports

#### Get all reports for a patient
```
GET /reports/patients/:patientId
```
**Response**: 200 OK - Array of reports for the patient
**Parameters:**
- `patientId` (path parameter) - Patient ID to filter reports

**Use Case:** Retrieve all medical reports associated with a specific patient

#### Get specific report by ID
```
GET /reports/:reportId
```
**Response**: 
- 200 OK - Report object
**Error**: 404 Not Found - If report doesn't exist

#### Create new report
```
POST /reports
Content-Type: application/json

{
  "patient": "patient_id",
  "doctor": "doctor_id",
  "report_content": "content_id"
}
```
**Response**: 201 Created - Report object with confirmation message
**Error**: 
- 400 Bad Request - Missing required fields (patient, doctor, report_content)
- 409 Conflict - Report with this content already exists

**Required Fields:**
- `patient` (ObjectId) - Patient reference
- `doctor` (ObjectId) - Doctor reference
- `report_content` (ObjectId) - Report content reference

#### Delete report
```
DELETE /reports/:reportId
```
**Response**: 200 OK - Deletion confirmation
**Error**: 404 Not Found - If report doesn't exist

**Use Case:** Remove outdated or incorrect medical reports from the system

## Key Business Logic

### Report Creation Validation

1. **Required Fields Check**: Validates that patient, doctor, and report_content are provided
2. **Duplicate Prevention**: Ensures report content is not duplicated in the system
3. **Referential Integrity**: Enforces ObjectId references to patient and doctor
4. **Error Handling**: Returns appropriate HTTP status codes with descriptive messages

### Response Codes

- `200 OK`: Successful GET or DELETE request
- `201 Created`: Report successfully created
- `400 Bad Request`: Missing required fields (patient, doctor, report_content)
- `404 Not Found`: Report or resource doesn't exist
- `409 Conflict`: Report content already exists
- `500 Internal Server Error`: Database or server error

## Performance Considerations

- **Indexes**: Ensure MongoDB has indexes on:
  - `Report.patient` (for patient-specific queries)
  - `Report.doctor` (for doctor-specific queries)
  - `Report.report_content` (for unique constraint and lookup)

- **Scaling**: This service is stateless and can be horizontally scaled behind a load balancer

- **Query Optimization**: Patient-based queries use indexed fields for fast retrieval

- **Archive Strategy**: Consider archiving old reports to MongoDB Atlas Archive storage for compliance

## Error Handling

The service provides descriptive JSON error responses:

```json
{
  "message": "Error description"
}
```

Common scenarios:
- Missing required fields returns 400
- Duplicate report content returns 409
- Report not found returns 404
- Database errors return 500

## Integration with Other Services

This service integrates with:

- **Patient Service** (Port 5003): Patient information and validation
- **Medical Folder Service** (Port 5002): Links reports to patient medical folders
- **Upload File Service**: Manages document storage for report content
- **Patient Interface Backend** (Port 5000): Exposes report APIs to frontend

## Data Flow

```
1. Doctor creates a report through Patient Interface Backend
2. Report Service receives the request
3. Validates patient, doctor, and content existence
4. Stores report in MongoDB
5. Report is linked to patient's medical folder
6. Report is indexed for fast retrieval
```

## Use Cases

### 1. Lab Report Creation
When a lab completes testing, a report is created by the doctor and linked to the patient.

### 2. Patient Medical History Review
Retrieve all reports for a patient to review their medical history and test results.

Store diagnostic reports from tests (X-ray, MRI, CT scan) in the system.

### 4. Clinical Notes
Archive clinical notes and observations from doctor consultations.

### 5. Report Retrieval for Consultation
Get a specific report to review during patient consultation.

### 6. Report Removal
Delete outdated or incorrectly entered reports from the system.
